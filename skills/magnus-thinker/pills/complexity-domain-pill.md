# Complexity Domain Pill

Inspirada en Cynefin: antes de resolver, clasificar el tipo de situacion. No todos los problemas aceptan la misma logica.

## Activar Cuando

- El usuario no sabe si necesita regla, experto, experimento o reaccion inmediata.
- Hay ambiguedad, caos, mercado incierto, problema social/humano o sistema con muchas variables.
- Una solucion "best practice" podria fallar por contexto.

## Dominios Operativos

| Dominio | Senal | Accion Magnus |
| --- | --- | --- |
| Claro | Causa-efecto obvia, regla conocida | Aplicar practica estandar, checklist, ejecucion rapida. |
| Complicado | Requiere analisis experto, pero hay respuesta | Diagnosticar, comparar expertos/herramientas, elegir buena practica. |
| Complejo | Causa-efecto se entiende despues, no antes | Probar pequenos experimentos, detectar patrones, escalar lo que funciona. |
| Caotico | No hay estabilidad, urge contener | Actuar para estabilizar, luego leer senales. |
| Confuso | No sabemos en que dominio estamos | Separar partes del problema y clasificar cada una. |

## Regla

No meter una "solucion experta" en un problema complejo sin experimento. No pedir experimento infinito en un problema claro.

## Output Util

- "Esto no es un problema claro; es complejo, conviene testear primero."
- "Esta parte es complicada pero resoluble con experto/herramienta."
- "Primero estabilizar, luego optimizar."

Fuentes: Cynefin.io y trabajos de Snowden/Kurtz sobre decision support y sense-making.
